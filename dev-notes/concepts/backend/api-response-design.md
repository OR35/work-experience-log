# API Response Design

## 목적
프론트엔드에서 일관되게 처리 가능한  
API 응답 구조를 유지하기 위함

---

## 기본 응답 구조

- success 여부
- data
- error 정보

---

## 설계 기준

- 성공 / 실패 응답 구조 통일
- 에러 메시지는 사용자용 / 로그용 분리
- 프론트에서 분기 처리 최소화

---

## 상태 코드 활용

- 200: 정상 처리
- 400: 잘못된 요청
- 401/403: 인증/권한 문제
- 500: 서버 내부 오류

---

## 실무 포인트

- HTTP Status + 응답 메시지 조합이 중요
- 프론트에서 예외 케이스를 예측 가능하게 해야 함
