# Technical Decisions

본 문서는 지능형 센서 플랫폼 개발 과정에서  
구조, 설계, 구현 방식에 대해 **의도적으로 선택한 결정 사항**을 정리한 문서입니다.

---

## Decision 1. 도메인 기준으로 기능 분리

### 선택 배경
초기에는 URL 또는 화면 단위로 기능을 구성했으나,
기능 확장 시 책임이 불명확해지는 문제가 발생했습니다.

### 고려 사항
- 기능의 목적과 책임을 명확히 구분할 필요
- 도메인별 독립적인 확장 가능성

### 결정
URL 구조가 아닌 **비즈니스 도메인 책임 기준**으로 기능을 분리했습니다.

### 결과
- 기능 확장 시 영향 범위 명확
- 문서화 및 커뮤니케이션 비용 감소

---

## Decision 2. 인증 / 권한 로직 중앙화

### 선택 배경
화면 단위로 권한 체크를 구현하면서
중복 로직과 보안 누락 가능성이 발생했습니다.

### 고려 사항
- 보안 로직의 일관성
- 유지보수 비용

### 결정
인증 및 권한 체크 로직을 공통 가드 및 미들웨어로 중앙화했습니다.

### 결과
- 권한 관련 버그 감소
- UI 및 API 단에서의 안정성 확보

---

## Decision 3. 공통 CRUD 패턴 도입

### 선택 배경
여러 관리 화면에서 유사한 CRUD 로직이 반복되었습니다.

### 고려 사항
- 개발 속도
- 코드 일관성

### 결정
목록/상세/등록/수정 흐름을 공통 패턴으로 정리했습니다.

### 결과
- 신규 화면 개발 시간 단축
- 유지보수성 향상

---

## Decision 4. 비동기 Job 상태 분리 관리

### 선택 배경
모델 학습·최적화는 즉시 결과를 반환할 수 없는 비동기 작업입니다.

### 고려 사항
- 사용자 경험
- 상태 동기화 문제

### 결정
Job 상태를 별도 관리하고 상태 기반 UI를 구성했습니다.

### 결과
- 사용자 혼란 감소
- 학습 연동 안정성 확보

---

## Decision 5. 운영 기능 분리

### 선택 배경
운영 로그와 비즈니스 로직이 섞이면서
코드 복잡도가 증가했습니다.

### 고려 사항
- 운영 안정성
- 장애 대응 속도

### 결정
로그/이력/모니터링 기능을 독립 도메인으로 분리했습니다.

### 결과
- 운영 기능의 독립성 확보
- 문제 추적 및 대응 속도 개선
